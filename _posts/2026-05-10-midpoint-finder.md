## 프로젝트 개요

다수의 출발지에서 가장 합리적인 모임 장소를 계산해 주는 서비스입니다. 별도의 앱 설치나 브라우저 접속 없이, 기존 토스(Toss) 앱 내의 **웹뷰(WebView) 환경**에서 구동되는 앱인토스(Apps in Toss) 미니앱으로 기획 및 개발 중입니다. 

단순히 웹페이지를 띄우는 것을 넘어, 사용자가 실제 네이티브 앱을 사용하는 것과 같은 매끄러운 경험을 제공하는 것에 초점을 맞췄습니다.

---

## 🛠 주요 개발 과정 및 도입 기술

### 1. 토스 디자인 가이드라인(Toss Design System) 적용
토스 앱 내부에서 실행되는 만큼, 기존 토스 UI와의 이질감을 없애는 것이 중요했습니다. 
- 토스 디자인 가이드라인을 참고하여 폰트 크기, 여백, 컴포넌트 형태를 구성했습니다.
- 페이지 이동(Routing)을 최소화하고 **바텀 시트(Bottom Sheet)** 중심의 인터페이스를 적용해 단일 화면 내에서 모든 주소 검색과 결과 확인이 이루어지도록 설계했습니다.

### 2. Cursor AI MCP를 활용한 개발 워크플로우
개발 생산성을 높이기 위해 Cursor 에디터와 **MCP(Model Context Protocol)** 기술을 적극 활용했습니다.
- AI가 프로젝트의 전체 폴더 구조와 의존성을 파악한 상태에서 컨텍스트 기반의 코드를 제안하도록 구성했습니다.
- 이를 통해 카카오맵 API 연동과 좌표 계산 로직 등 프론트엔드의 복잡한 구현 시간을 단축하고, 보일러플레이트 코드 작성을 최소화할 수 있었습니다.

### 3. 웹뷰(WebView) 보안 정책 대응: 지도 직접 렌더링
토스 내부망의 강력한 보안 정책으로 인해 일반적인 웹 개발에서 자주 쓰이는 `<iframe>` 방식의 지도 삽입이 차단되었습니다.
- 이를 해결하기 위해 카카오맵 Javascript SDK를 도입했습니다. 
- React의 `useRef`를 활용해 렌더링할 가상의 DOM 엘리먼트(`<div>`)를 잡고, 컴포넌트 마운트 시점에 해당 노드 위에 지도를 직접 그리는(Direct Rendering) 방식으로 코드를 재작성하여 보안 심사 기준을 충족했습니다.

### 4. 미니앱 번들 패키징 및 용량 최적화
웹뷰 로딩 속도 확보를 위한 앱인토스의 배포 규격을 맞추는 작업을 진행했습니다.
- 전체 소스 코드가 아닌 빌드 결과물(`dist` 폴더)만을 독립적으로 패키징하도록 배포 프로세스를 분리했습니다.
- 불필요한 에셋을 줄여 최종 미니앱 번들(`.ait`) 용량을 **558KB**로 경량화하여 배포했습니다.

## 💡 회고

토스 앱이라는 특수한 웹뷰 환경을 타겟으로 개발하면서, 플랫폼이 요구하는 디자인 시스템과 보안 정책(iframe 차단 등)에 맞춰 프론트엔드를 최적화하는 과정을 거쳤습니다. 특히 Cursor AI의 MCP를 활용하여 새로운 형태의 효율적인 개발 워크플로우를 실제 프로젝트에 적용해 볼 수 있었던 유의미한 경험이었습니다.
